# Data

All files in this directory are the inputs consumed by the experiments under
`experiments/`. They are static facts (synthetic task graphs and real lunar
terrain rasters) — no training-time logs or model checkpoints live here.

## Schema overview

Every scenario JSON (training, OOD, and DEM-grounded) shares the same top-level
shape:

```
{
  "taskgraph": {
    "version": "0.1",
    "mission_id": "<string>",
    "nodes":   [ { id, op_type, params, location, estimated_duration_s,
                   agent_type_reqs, capability_reqs, produces, requires_data,
                   requires_connected, time_window, resources, ... }, ... ],
    "edges":   [ { id, type ("order"|"data"|"mutex"), src, dst,
                   resource, milestone, note }, ... ],
    "constraints": { min_battery_percent, max_concurrent_tasks_per_agent,
                     periodic },
    "metadata": { ... per-graph notes ... }
  },
  "metadata": {
    "scale":       "small" | "medium" | "large" | "ood",
    "variant_description": "<human-readable note>",
    "generated_by": "<generator name>",
    "node_count":  <int>,
    "edge_count":  <int>,
    "agent_count": <int>,
    "baseline_method": "greedy+CPM" | ...
  },
  "baseline_sim_result": { makespan, task_completion_rate, energy_consumption,
                           constraint_violations, comm_connectivity_rate,
                           bottleneck_tasks, completed_tasks, failed_tasks,
                           greedy_cpm_makespan, greedy_coupled_makespan,
                           cpsat_makespan, cpsat_status, cpsat_wall_s },
  "baseline_schedule":   { mission_id, makespan_s,
                           items: [ { node_id, agent, start_s, end_s }, ... ] }
}
```

`baseline_sim_result` / `baseline_schedule` are pre-computed reference outputs
from the greedy-CPM scheduler on a default agent fleet; experiments that train
or evaluate alternative schedulers ignore these blocks and recompute. They are
shipped here so that the corpus can be sanity-checked offline without running
the simulator.

## `training_corpus/` — 14 in-distribution scenarios

Hand-crafted by the tool-library generator to span the operational regimes that
the lunar mission planner is expected to handle. Grouped by graph size:

**Small (n = 3)** — single rover + relay, 5–10 tasks. Trains base behaviours.

| File | Variant |
|---|---|
| `small_a_pure_coverage.json`     | A: pure coverage (relay + 4-cell grid, no sampling) |
| `small_b_scan_and_sample.json`   | B: scan + sample, single sampler |
| `small_c_multi_sample.json`      | C: multi-sample site, capability mix |

**Medium (n = 3)** — multi-agent fleets, 10–40 tasks. Probes parallelism.

| File | Variant |
|---|---|
| `medium_a_deep_sector.json`      | A: deep sector exploration |
| `medium_b_dual_parallel.json`    | B: two parallel branches |
| `medium_c_relay_chain.json`      | C: relay-chain backbone |

**Large (n = 8)** — full fleets, 26–84 tasks. Stresses scheduler under
heterogeneous structural regimes.

| File | Variant |
|---|---|
| `large_a_full_coverage.json`     | A: full coverage |
| `large_b_high_mutex.json`        | B: high mutex density |
| `large_c_deep_chain.json`        | C: deep chain |
| `large_d_wide_parallel.json`     | D: wide parallel |
| `large_e_mixed_dag.json`         | E: mixed DAG |
| `large_f_deep_mutex.json`        | F: deep mutex chain |
| `large_g_relay_tree.json`        | G: relay tree |
| `large_h_burst_sampling.json`    | H: burst sampling |

## `training_corpus_ood/` — 40 out-of-distribution scenarios

Procedurally generated by the OOD generator. Five topology classes × 8 seeds
each. Used to evaluate generalization of trained schedulers beyond the
hand-crafted training distribution.

| Class file prefix | Class | Count |
|---|---|---:|
| `ood_deep_chain_NNN.json`     | Deep dependency chains (high longest-path)            | 8 |
| `ood_mixed_NNN.json`          | Mixed DAGs with branching + recombination             | 8 |
| `ood_mutex_dense_NNN.json`    | Dense mutual-exclusion edges                          | 8 |
| `ood_parallel_pure_NNN.json`  | Wide parallel fan-out, minimal coupling               | 8 |
| `ood_sampling_burst_NNN.json` | Bursty sampling phases                                | 8 |
| **Total**                     |                                                       | **40** |

OOD scenarios use the same top-level schema as the training corpus but with
graph sizes typically larger (≈ 100–200 nodes) and an additional
`metadata.structural_features` block summarizing `n_tasks`, `n_edges`,
`mutex_edges`, `mutex_ratio`, `longest_chain_depth`, `parallelism_index`.

## `dem/` — Lunar terrain (Shackleton crater rim)

Source: NASA LOLA south-polar DEM (`LDEM_83S_10MPP_ADJ.tiff`, 10 m/px).
Sample point: 340° azimuth, 10 km radius from the south pole — corresponds to
the Shackleton crater rim (≈ −89.65° lat, 222.85° lon).

| File | Size (KB) | Description |
|---|---:|---|
| `shackleton_1km.npz`           | 65 | 100 × 100 grid, 10 m cells (1 km × 1 km region) |
| `shackleton_5km.npz`           | 65 | 100 × 100 grid, 50 m cells (5 km × 5 km region) |
| `shackleton_terrain.npz`       | 4  | Small smoke-test fixture (legacy 200 m × 200 m crop) |
| `real_shackleton_1km.json`     | 65 | Matching task graph for the 1 km tile (78 tasks) |
| `real_shackleton_5km.json`     | 68 | Matching task graph for the 5 km tile (220 tasks) |

Each `.npz` archive stores the following arrays:

| Array         | Shape    | Dtype  | Units / meaning |
|---------------|----------|--------|-----------------|
| `elevation`   | (H, W)   | float  | metres above local reference (negative = below mean lunar radius) |
| `slope`       | (H, W)   | float  | degrees |
| `is_psr`      | (H, W)   | bool   | permanently-shadowed region mask (illumination-derived) |
| `is_obstacle` | (H, W)   | bool   | `slope > 30°` mask used by the lunar simulator |
| `cell_size`   | scalar   | float  | grid cell edge length in metres (10.0 or 50.0) |

### Physical parameters

| Parameter         | Value             | Notes |
|-------------------|-------------------|-------|
| Cell size         | 10 m (1 km) / 50 m (5 km) | Spatial resolution of slope + PSR |
| Comm range        | 500 m (1 km) / 700 m (5 km) | Drives relay placement |
| Agent speed       | 0.5 m/s           | LunarSimulator default; ~30 min to traverse 1 km |
| Slope threshold   | 30°               | Above which `is_obstacle = True` |

### Notes on the 5 km scenario

The default agent energy budget (8000 Wh) is insufficient for the 220-task
5 km mission. Single-task completion rate hovers around 40 % under the
out-of-the-box `make_agents()` defaults. To exercise the 5 km scale
productively, either raise `energy_wh` in
`experiments/<scheduler>/common.py:make_agents()` (e.g. 20 000 Wh), reduce the
rover-only task ratio, or augment the simulator with a return-to-base recharge
behaviour. The 1 km scenario is the recommended default for reported
benchmarks; 5 km is reserved as a stretch case.
